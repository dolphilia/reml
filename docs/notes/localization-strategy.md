# ローカライゼーションおよび国際化戦略案

このドキュメントは、Reml プロジェクトにおけるローカライゼーション（L10n）と国際化（i18n）の戦略案をまとめたものです。
開発者の母語である日本語を活かした開発効率と、グローバルな利用者層へのリーチを両立させることを目的としています。

## 1. 基本方針

- **開発プロセス（Internal）**: 日本語主体。思考の速度と精度を優先する。
- **公開成果物（Public）**: 多言語対応（英語を最優先）。グローバルスタンダードに合わせる。
- **コミュニティ（Communication）**: 日本語ファーストだが、英語も歓迎する「バイリンガル・フレンドリー」な体制。

## 2. リポジトリ管理とドキュメント構成

### 2.1 ファイル構造と命名規則

現状通り、**ディレクトリ構造やファイル名は英語（kebab-case）** で統一します。これは多くの開発ツールやOSでの互換性、および非日本語話者が構造を把握しやすくするためです。

### 2.2 README.md (ルート)

リポジトリの顔となる `README.md` は、**英語をメイン** に据えることを推奨します。
ただし、開発者の想いや日本のコミュニティへの配慮として、以下のいずれかのアプローチを採ります。

*   **案A（推奨）**: 英語の `README.md` を配置し、冒頭に「[🇯🇵 日本語版はこちら](./README_ja.md)」というリンクを目立つように置く。
*   **案B**: `README.md` 内でセクションを分け、英語と日本語を併記する（メンテナンスコストが若干高い）。

### 2.3 ドキュメントの分類と言語戦略

ドキュメントを「内部用」と「公開用」に明確に区別し、言語リソースの配分を最適化します。

| 分類 | 対象ディレクトリ | 言語方針 | 理由 |
| :--- | :--- | :--- | :--- |
| **内部資料** | `docs/plans/`, `docs/notes/` | **日本語のみ** | 開発者の思考・計画プロセスであり、翻訳コストをかけず速度を優先する。必要に応じてDeepL/LLM等で読むことは可能。 |
| **仕様書** | `docs/spec/` | **日本語原本 + 英語翻訳** | 言語仕様は厳密性が求められるため、原本は日本語で管理し、確定後に翻訳する（または英語を正とする運用に移行する将来を見据える）。 |
| **ガイド** | `docs/guides/` | **多言語推奨** (Ja, En, etc) | ユーザーが最初に触れる部分であり、ローカライズの効果が最も高い。 |
| **API/Code** | コード内コメント | **日本語主体** | 複雑なロジックの説明は日本語で行う。パブリックAPIのdoc commentのみ英語併記、または生成時に翻訳するフローを検討。 |

## 3. ドキュメントサイトの構築 (Docs Site)

公式ドキュメントサイト（例: `reml-lang.org`）は、**ローカライズ前提の静的サイトジェネレーター** を採用します。

### 推奨ツール: Docusaurus または VitePress

これらは優れたi18n機能を標準で備えています。

*   **構成案**:
    *   `docs/ja/` (Source of Truth): 日本語で執筆。
    *   `docs/en/` (Translation): 日本語から翻訳。
    *   その他の言語 (zh, es, pt): 必要に応じてコミュニティベースで翻訳を追加。

### ワークフロー

1.  コンテンツは `docs/ja` 配下のMarkdownとして日本語で執筆・コミットする。
2.  リリースや大きな更新のタイミングで、翻訳ツール（AI補助含む）を用いて `docs/en` を更新する。
3.  CIでリンク切れ等をチェックする。

## 4. CLIツール・コンパイラのローカライズ

現在、CLIのメッセージは日本語と英語が混在、あるいは「日本語（英語キーワード）」という形式になっています。これを整理し、ユーザーの環境に合わせた表示を実現します。

### 4.1 メッセージカタログの導入

ソースコード内にハードコードされている文字列を抽出し、ID管理します。

*   **現状**: `Printf.sprintf "挿入 [%s]: '%s'"`
*   **改善案**: `Messages.get_formatted MessageID.FixitInsert [span; text]`

### 4.2 実装方針

*   **デフォルト言語**: **英語**
    *   多くの開発者は英語のエラーメッセージに慣れており、検索性（ググりやすさ）の観点からも英語が有利です。
*   **日本語対応**: 環境変数（`LANG=ja_JP.UTF-8` や `REML_LANG=ja`）を検出して切り替える。
    *   または、現状の「日本語（英語キーワード）」スタイルを "Verbose Mode" や "Beginner Mode" として残すことも検討できますが、基本は言語切り替えが望ましいです。
*   **Rust移行を見据えて**:
    *   Rustには `fluent-rs` (Project Fluent) や `gettext–rs` などの成熟したエコシステムがあります。移行時にこれらを採用することを前提に、OCaml版では簡易的な `Msg` モジュールで Key-Value 対応させる程度でも十分です。

## 5. 開発コミュニティとコミュニケーション

### 5.1 Issue / Pull Request

*   **方針**: 「**日本語ファースト、英語ウェルカム** (Japanese First, English Welcome)」
*   **対応**:
    *   Issueテンプレートに「You can write in English or Japanese」と明記する。
    *   海外からのIssueに対しては、無理に英語で返信しなくても、翻訳ツールを使いながら日本語で返信しても良い（最近のGitHubには翻訳機能も統合されつつある）。
    *   重要なリリースノートやBreaking Changeのアナウンスのみ、英語併記を必須とする。

### 5.2 コミットメッセージ

*   こちらも開発者のパフォーマンスを優先し、**日本語許容** とするのが良いでしょう。
*   ただし、将来的に海外のコアコントリビューターが増えた段階で英語必須に切り替える（Rust移行後のフェーズ3-4あたりを目処）のが現実的です。

---

以上の戦略により、開発者の「日本語での思考・開発」を阻害せず、かつ「グローバルな利用者に受け入れられる」プロダクトを目指します。
