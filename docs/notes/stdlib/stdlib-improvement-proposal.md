# 標準ライブラリ改善提案：DSL開発者体験の向上

## 1. 概要
Reml は、開発者がドメイン特化言語（DSL）を効率的に構築できるようにする「DSLファースト」な言語を目指しています。現在の標準ライブラリ（`Core.Parse`、`Core.IO` など）は強力なプリミティブを提供していますが、テスト、ドキュメント作成、CLI 作成、フォーマットといった、DSL の *ライフサイクル全体* を支援する高レベルなツールにはギャップがあります。

本提案では、このギャップを埋めるために、現代的なエコシステム（Rust、OCaml、Haskell）に触発された新しいモジュールと機能について概説します。

## 2. 提案モジュール

### 2.1 `Core.Test` — 統合テストとゴールデンテスト
**インスピレーション**: Rust (`cargo test`, `insta`), Haskell (`tasty-golden`), OCaml (`dune runtest`)

**現状のギャップ**: テストは現在、低レベルのアサーションやシェルスクリプト（`run_examples.sh`）に依存しています。DSL には、パーサーのエッジケースの厳密なテストや出力検証が必要です。

**提案**:
- **`Core.Test` モジュール**: 標準テストライブラリ。
- **スナップショット/ゴールデンテスト**: `assert_snapshot!(parsed_ast)` のファーストクラスサポート。複雑な AST の等価性を手動で記述するのは面倒であるため、パーサーのテストには不可欠です。
- **パラメータ化テスト**: テーブル駆動テスト（入力 -> 期待される出力）の容易な定義。
- **ファジングインターフェース**: パーサーの堅牢性（OOM、無限ループ）をテストするために入力（ランダムなバイト列/テキスト）をファジングするための標準フック。

```reml
use Core.Test

test "parser should handle nested list" {
  let input = "[1, [2, 3]]";
  let ast = MyDsl.parse(input)?;
  assert_snapshot!(ast);
}
```

### 2.2 `Core.Cli` — 宣言的コマンドラインインターフェース
**インスピレーション**: Rust (`clap`), Haskell (`optparse-applicative`), OCaml (`cmdliner`)

**現状のギャップ**: ユーザー定義の DSL は、独自の CLI ツール（コンパイラ、インタプリタ）を必要とすることがよくあります。現在、引数の解析には手動の実装や `Core.Env` の操作が必要です。

**提案**:
- **宣言的ビルダー**: 構造ベースの CLI 定義（フラグ、引数、サブコマンド）。
- **ヘルプ自動生成**: 使用法/ヘルプテキストの自動生成。
- **型安全な解析**: 引数を Reml レコードに直接解析。

```reml
use Core.Cli

type MyCliArgs = {
  verbose: Bool,
  input: Path,
}

let cli = Cli.builder()
  .flag("verbose", help="詳細なログ出力を有効にする")
  .arg("input", help="入力ソースファイル")
  .build();
```

### 2.3 `Core.Text.Pretty` — Wadler-Leijen プリティプリンタ
**インスピレーション**: Haskell (`prettyprinter`), OCaml (`pprint`), JS (`prettier` アルゴリズム)

**現状のギャップ**: DSL はコードを出力（トランスパイラ）したり、それ自体をフォーマット（オートフォーマッタ）したりする必要があります。文字列連結や単純なテンプレート（`Core.Text.Template`）では、複雑なインデントロジックには不十分です。

**提案**:
- **コンビネータ**: `text`, `line`, `nest`, `group`, `softline`。
- **レイアウトチェック**: ページ幅に基づいて、改行とインデントを自動的かつ効率的に処理します。
- **統合**: AST ノードの `Format` トレイトを簡単に実装するためのヘルパー。

これにより、ユーザーは最小限の労力で *独自の* DSL 用に `reml fmt` スタイルのフォーマッタを作成できるようになります。

### 2.4 `Core.Doc` — ドキュメントジェネレータ
**インスピレーション**: Rust (`rustdoc`), Javadoc

**現状のギャップ**: Reml 自体には `reml doc` がありますが、DSL 作者は *自身の* ライブラリや DSL 構成要素のドキュメントを作成する必要があります。

**提案**:
- **ドキュメントコメント抽出**: ソースファイルからドキュメントコメント（`///`）を解析する API。
- **静的サイト生成**: これらを HTML/Markdown にレンダリングするユーティリティ。
- **テスト可能な例**: 「Doctest」のサポート — ドキュメント内のコードスニペットを実行して、最新の状態に保たれていることを確認します。

### 2.5 `Core.Lsp` — LSP サーバーツールキット
**インスピレーション**: `tower-lsp` (Rust), `ocaml-lsp` ライブラリ

**現状のギャップ**: 「DSLファースト」のビジョンは、ユーザーがカスタム言語に対して IDE サポートを得られることを意味します。LSP サーバーをゼロから作成するのは困難です。

**提案**:
- **LSP プロトコル型**: `Initialize`, `TextDocumentDidChange`, `Completion` などの標準定義。
- **サーバのスキャフォールド**: `Core.IO` と LSP JSON-RPC メッセージループを接続するためのヘルパー。
- **インテリジェンス用コンビネータ**: 「カーソル位置の単語」の検索や単純な補完リストを実装するためのヘルパー。

## 3. Phase 4/5 に向けた優先度

| 機能 | 優先度 | 正当性 |
| :--- | :--- | :--- |
| **`Core.Test`** | **高** | Rust 移行の正確性とユーザー DSL の正しさを直ちに保証するために不可欠。 |
| **`Core.Cli`** | **高** | DSL 用のコンパイルツール（主要なユースケース）を構築するために必要。 |
| **`Core.Text.Pretty`** | **中** | コード生成に重要だが、当面はテンプレートで代用可能。 |
| **`Core.Lsp`** | **中** | 価値は高いが複雑度も高い。基本的なツールの後に着手可能。 |
| **`Core.Doc`** | **低** | エコシステムが成長するまで後回し可能。 |

## 4. 結論
これらのモジュールを追加することで、Reml は「パーサーを書くのに関連する言語」から「言語を作成するための包括的なツールキット」へと移行します。これは、「DSLファースト」な開発を実現するというプロジェクトの目標と完全に一致しています。
