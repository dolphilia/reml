# Reml コンパイラ移植原則（統合版）

本書は Reml コンパイラを OCaml 実装から Rust 実装へ移行する際に参照する総合ガイドである。一般的な言語移植のセオリー、OCaml→Rust の設計上の注意点、そして Reml プロジェクト固有の要件を一体化し、他文書に依存せず単独で利用できるようにまとめている。

## 1. 基本姿勢と成功指標
- **優先順位原則**: 振る舞いの同一性 > 設計の同一性 > 実装の同一性 > 文法の同一性。見た目より観測可能な挙動を最優先する。
- **移植目的**: Windows x64 (MSVC/GNU) での再現性確保、LLVM バインディング依存からの脱却、セルフホスト移行への準備、高安全性な所有権モデルへの移行。
- **主な成功指標**:
  - 仕様サンプル・ゴールデンテスト通過率 ≥ 95%。
  - 診断・監査キー（`effect.stage.*`, `bridge.stage.*` など）が完全一致。
  - GitHub Actions `windows-latest` × `msvc`/`gnu` マトリクスで 5 連続スイートグリーン。
  - 代表ベンチマーク（現行 OCaml 実装）比で性能回帰 ±10% 以内。
  - 仕様書・計画書への脚注更新と `docs-migrations.log` 記録の完了。
- **非目標**: 初期段階で完全な最適化を求めない。可観測な等価性と安全性を確保した後に最適化を行う。

| P1: フロントエンド移植 | パーサ／型推論／診断前処理の Rust 移植と差分検証 | Rust AST/IR、`Diagnostic` 出力互換ログ、`inkwell` PoC |

| P3: CI/監査統合 | 差分ハーネス本格運用、監査メトリクス共有、Windows 完全移行 | GitHub Actions 更新、監査メトリクス連携、診断ゴールデン差分レポート |

- **ハイブリッド運用**: Rust 実装が完成するまで OCaml 実装と二重出力（dual-write）で比較。FFI ブリッジを使い旧実装を呼び出すことで差分を可視化する。各フェーズの dual-write 期間は最大 2 スプリントを目安にし、完了後は OCaml 実装を凍結。
- **アダプタ層先行**: FS/ネット/時刻/乱数/暗号/JSON/DB など環境依存の差異は薄いラッパで吸収し、差分テストハーネスと観測指標を早期に整備する。

## 3. 型・データモデル設計
- **整数と数値**: OCaml `int` は 31/63bit タグ付き。用途に応じ `isize`（プラットフォーム依存）または `i64`（仕様安定）を採用し、`wrapping_*`/`checked_*` の利用規約を定義。`Int32`/`Int64`/`Nativeint` は幅固定で 1:1 対応。
- **浮動小数**: OCaml `float` は倍精度（`f64`）。丸めや NaN 取扱いは観測差分として固定。
- **文字列とバイト列**: OCaml `string` は実質バイト列。Rust ではバイト指向を `Vec<u8>`/`&[u8]`、テキストを `String`/`&str` に分離。OCaml `char` の 8bit と Rust `char` の Unicode スカラ差に注意。
- **代数的データ型（ADT）**: OCaml バリアント → Rust `enum`。再帰構造は `Box<T>`/`Rc<T>` 等で明示。未完拡張を想定するものは `#[non_exhaustive]`。
- **GADT/多相**: GADT は `enum` + `PhantomData` またはトレイト階層で表現。ポリモーフィックバリアントは `enum` に正規化し、開放集合が必要なら sealed trait パターン。
- **レコード**: OCaml レコード → Rust 構造体。公開フィールドには `pub`、デフォルト値は `Default` 実装。ラベル順の意味を保つ。
- **遅延評価**: `Lazy.t` → `once_cell`/`LazyLock` または明示キャッシュ。
- **再帰的 AST/IR**: `Box<T>`/`Vec<T>`/arena を組み合わせ、割り当て制御と共有のバランスを取る。

## 4. 制御フロー・エラー・所有権
- **例外→Result**: 例外は `Result<T, E>`/`Option<T>` に正規化。エラー型は規模に応じて `enum` → `thiserror` → `anyhow`/独自階層を選択。`?` 演算子で伝播。
- **非局所ジャンプ**: パーサのバックトラックなどは戻り値に `Backtrack`/`Committed` 状態を持たせる。
- **末尾再帰**: Rust は TCO を保証しない。深い再帰はループ/明示スタックに置換。
- **所有権/共有**: 選定基準を明確化。
  - 単一所有→所有権移動。
  - 読み取り共有→`Rc<T>`/`Arc<T>`（スレッド跨ぎは `Arc`）。
  - 読み書き共有→`Rc<RefCell<T>>`/`Arc<Mutex<T>>`/`Arc<RwLock<T>>`。
  - 一時アクセス→`&T`/`&mut T`。
  - 循環参照→`Weak<T>` でリーク防止。
- **`Cow` の活用**: ホットパスのコピー削減に `Cow<'a, T>` を検討。

## 5. パターンマッチとモジュール体系
- **パターンマッチ**: 網羅性は `match` で保持。将来拡張は `_` を許容しテストでケース表を固定。所有権維持が必要な場合は `match &expr` で参照マッチ。
- **ガード**: 条件は副作用を避け純粋に。Rust の `if` ガードで表現。
- **モジュール/ファンクタ**: OCaml モジュール/ファンクタ → Rust モジュール/ジェネリクス/トレイト。第一級モジュールは trait object (`dyn Trait`) で代替し、ディスパッチ箇所は性能測定。
- **公開 API 管理**: `.mli` に相当する公開仕様は `pub`/`pub(crate)` で制御。Rust 2018 のフラットなファイル構成を採用し、サブモジュールは `mod sub;` とファイル分割で表現。

## 6. 並行性・非同期性
- **スレッドモデル**: Rust の `Send`/`Sync` を境界に設定。共有状態は `Arc<Mutex<T>>` 等で管理し、`Send` でない型の拡散を禁止。
- **チャネルとタスク**: `crossbeam` や標準チャネルを活用。OCaml の `Mutex`/`Condition` は `Arc<Mutex<T>>` + `Condvar` に移植。
- **非同期**: OCaml の Lwt/Async を Rust `async/.await` + ランタイム（tokio/async-std）で再構成。境界 API は同期ラッパを提供。

## 7. Windows / 環境差分対策
- `rustup` で MSVC/GNU 双方のツールチェーンを管理し、手動スクリプト（`reml-msvc-env`）依存を排除。
- CRLF、ファイル名の大文字小文字、Unicode 正規化、タイムゾーン、ロケール差分を観測項目として固定。
- Windows 固有の DLL エクスポート（`dllimport`/`dllexport`）や `/MT` `/MD` のビルド方針を明文化。
- 環境診断スクリプト（Rust 版）を `tooling/toolchains/` に用意し、CI で定期実行。

## 8. LLVM / FFI / unsafe 境界
- `inkwell`/`llvm-sys` を評価し、バージョン固定と nightly 検証環境を整備。`TargetMachine`/`DataLayout`/`CallingConv` は既存 OCaml 実装と照合。
- `opt -verify` を CI ゲート化。生成 IR の整合を自動検証。
- FFI では `repr(C)` を使用し、`static_assertions` 等でサイズ/アラインメントを検証。`extern "C"` で呼出規約を明示。
- `unsafe` ブロックは安全ラッパで包み、境界を局所化。レビュー時に根拠と検証手順を記録。

## 9. テスト・検証・可観測性
- **テスト階層**:
  - 等価性テスト: 旧実装と新実装の入出力をゴールデン化して比較。
  - 契約テスト: 公開 API/CLI/JSON を差分ハーネスで検証。
  - プロパティテスト: パーサ再構成、型推論の主性など不変条件を proptest/QuickCheck 系で確認。
  - 診断テスト: エラー位置・期待/実際・提案（did-you-mean）。
  - ベンチマーク: `criterion` 等で代表ケースを測定し、許容回帰範囲を合意。
  - メタモルフィック/環境差分テスト: 入力変換や CRLF/TZ の違いで挙動不変を確認。
- **観測基盤**:
  - `collect-iterator-audit-metrics.py` など既存スクリプトを Rust CLI に対応させる。
  - ログ/トレース ID/メトリクス/ダンプ形式を OCaml 実装と共有し、CI で自動収集。
  - dual-write `diff-harness` に `syntax:expr-*` トレース（`reports/spec-audit/ch1/effect_handler-20251118-trace.md` 等）を取り込み、Module/Expr 両方の観測点を固定する。
- **テスト配置**: ユニットテストは `#[cfg(test)]`、統合テストは `tests/` ディレクトリで管理。共通治具は `tests/common/`。

## 10. ビルド/CI/リリースフロー
- dune から Cargo へ移行し、機能フラグ (`features`) で構成を管理。Cross-compilation や Stage 切替は Cargo profile に統合。
- GitHub Actions で Linux/macOS/Windows × x86_64/arm64 × MSVC/GNU をマトリクス化。リンクチェック、診断 JSON スキーマ、IR 検証をジョブに組み込む。
- 差分テストハーネスを自動化し、出力差分はレビュー承認（理由付き）を必須化。
- 可観測性（ログ/メトリクス）と CI を結びつけ、回帰検出時に自動エスカレーション。

## 11. リスク管理と緩和策
- **Windows 向け LLVM 連携失敗**: バージョン固定・nightly 検証・C FFI の暫定回避ルートを確保。
- **診断キー逸脱**: 差分テストを必須とし、逸脱は即リスク登録。修正まで dual-write を維持。
- **フォーマット差分**: 観測差分チェックリスト（CRLF/Unicode/TZ）を遵守し、プラットフォーム別ゴールデンを用意。
- **チーム学習コスト**: 重要モジュールごとに PoC とレビューガイドラインを整備。研修進捗を技術的負債リストで追跡。
- **dual-write 長期化**: フェーズごとに期限を設定。期限超過時は原因分析と追加リソース投入を検討。

## 12. チェックリスト
- **着手前**
  - 型/API 対応表、数値/文字列/Unicode ルール、末尾再帰置換方針、例外→Result 変換規則を確定。
  - Windows ツールチェーン診断（OCaml/Rust）の比較結果を取得。
  - 診断・監査ゴールデンテストを Rust で再生し、差分を記録。
- **スプリントごと**
  - 差分テスト・プロパティテスト・`opt -verify` を実行し、ベンチ結果を監査ログへ追記。
  - Windows CI (MSVC/GNU) と監査メトリクススクリプトを最低 1 回実施。
  - 仕様・ガイド・ノート更新、`docs-migrations.log` 追記を確認。
- **マージ前**
  - `repr(C)`/サイズ/ABI チェック、`unsafe` レビュー、公開 API の `rustdoc` ドキュメント確認。
  - 変更理由を関連計画書へ脚注として明記し、リスク・TODO を登録。
- **リリース前/フェーズ移行前**
  - Windows 対応未完項目のクローズ記録、計画書目次更新、Phase 3+ ハンドオーバー資料に移植状況を反映。
