# WS6: 左再帰対処（Left Recursion）計画（ドラフト）

## 背景と狙い
式文法など多くの言語文法は左再帰を含むが、一般的な PEG/コンビネーターは左再帰を素直には扱えない。
調査メモ `docs/notes/core-parse-improvement-survey.md` でも「左再帰ガード」の可能性に言及している。

Reml では `chainl1` などの典型回避策に加え、OpBuilder/優先度ビルダー（`docs/spec/2-4-op-builder.md`）との整合も重要となる。

## 目標（ドラフト）
- 左再帰を含む文法を「落とし穴を避けて」書くための実用ガイドがある
- 代表的な式文法が、優先度/結合性を含めて安定に実装できる
- 退行（無限再帰/極端な遅さ）を回帰で検出できる

## 取り得る方針（ドラフト）
- **方針A: ガイドライン重視**（推奨）
  - 左再帰は `chainl1/chainr1` または `expr_builder` 相当へ変換して書く
  - 必要なら「左再帰ガード」を補助として提供（仕様外の補助でも可）
- **方針B: 自動左再帰対応**（保留）
  - Packrat と組み合わせて自動化する案は強力だが、仕様・実装の複雑度が高い

## タスク分割（ドラフト）
### Step 1: 仕様と既存計画の位置づけ確認
- `docs/spec/2-6-execution-strategy.md` と `docs/spec/2-4-op-builder.md` の記述を読み合わせ、左再帰の扱い（対応/非対応/補助）を明確化する
- bootstrap-roadmap の演算子ビルダー計画（Phase8 相当）の成果物と重複しないよう、ここでは「ガイドと回帰」を中心に整理する

### Step 2: 式サンプルの確定
- 優先度・結合性・括弧・単項/二項を含む最小式言語をサンプル化する
- Cut/Label（WS1/WS2）と組み合わせ、式エラーの期待が自然に出ることを確認する

### Step 3: 回帰項目の追加
- 無限再帰が起きる入力を「失敗として扱える」ことを検証する（診断キー設計は要検討）
- 極端に遅くなる入力を作り、観測フック（プロファイル）と合わせて監視する

## リスクと緩和
- 自動左再帰対応は実装負債になりやすい  
  → まずはガイドライン + 優先度ビルダー（既存仕様）で実用域へ到達させる
