# フェーズ 3: 意味解析 (Sema)

このフェーズでは、意味的検証、名前解決、型チェック、効果の基礎解析を実装します。

## 3.0 前提と成果物
- **前提**: フェーズ 2 の AST にスパン情報があり、最低限の構文エラーは排除済み。
- **成果物**:
  - 解決済み識別子を参照できる AST (Symbol ID / Type ID を付与)。
  - 型・効果の診断を出せるエラーレポータ。
  - 後続フェーズで利用できる「型付き AST」または「型注釈付き AST」。

## 3.1 シンボルテーブルとスコープ管理
- **データ構造**: `UTHASH` ベースのシンボルハッシュマップ。
- **スコープ**: デックのスタック、または永続データ構造アプローチ。
- **タスク**:
  1.  `Symbol` 構造体 (種類: Func, Var, Type, Module) の定義と `SymbolId` の導入。
  2.  値名前空間と型名前空間の分離 (同名の型/値を許容)。
  3.  `enter_scope()` と `exit_scope()` の実装。
  4.  シャドーイングをサポートする `lookup_symbol()` の実装。
  5.  `builtin`/`prelude` の登録と再定義禁止のルールを明文化。

## 3.2 名前解決
- **目標**: AST 識別子をその定義にリンクする。
- **タスク**:
  1.  宣言の収集 → 参照解決の 2 パス方式 (モジュール単位)。
  2.  AST ウォーク (C での Visitor パターン実装)。
  3.  変数/型/モジュールパスの解決。
  4.  「未定義の識別子」「重複定義」の診断報告。
  5.  モジュール/インポートの処理 (基本的なファイル検索と循環検出)。

## 3.3 型システムの表現
- **構造体**: `include/reml/typeck/` 内の `Type` enum/union。
  - プリミティブ: `Int`, `Float`, `Bool`, `Char`, `Unit`.
  - 複合: `Struct`, `Enum`, `Tuple`, `Function`, `Array`, `Ref`.
  - 変数: `TypeVar` と `Scheme` (ランク 1 の多相)。
- **メモリ**: ポインタ比較による等価性チェックのための型インターニング (Arena または Hash Consing)。
- **ライブラリ**: インターニングテーブル用の `uthash`。
- **備考**: 関数型に効果集合を保持し、後続の効果解析と統合する。

## 3.4 型チェック (推論と検査)
- **仕様**: `docs/spec/1-2-types-Inference.md`。
- **戦略**: 双方向型検査 (Bidirectional Type Checking) を基本に、必要に応じて単一化を併用。
- **タスク**:
  1.  `unify(t1, t2)` の実装 (occurs check 含む)。
  2.  `infer(expr) -> Type` の実装。
  3.  `check(expr, expected_type)` の実装。
  4.  `let` 一般化とインスタンス化 (ランク 1) の導入。
  5.  型注釈の尊重とエラー品質向上 (期待型との差分報告)。
  6.  リテラルの多相性処理 (例: `1` は `Int` または `Float` または `BigInt`)。

## 3.5 効果分析 (予備的)
- **仕様**: `docs/spec/1-3-effects-safety.md`。
- **目標**: 関数境界での基本的な効果 (IO, State, Panic) の追跡。
- **タスク**:
  1.  効果注釈と属性 (`@pure`, `@no_panic`) の解析。
  2.  式レベルの効果集合を計算し、関数型に反映。
  3.  コールグラフ上の効果伝播 (ナイーブな実装)。
  4.  値制限の判定に効果情報を利用する。

## 3.6 診断とエラーメッセージ
- **目標**: 位置情報 (行/列) と原因を明示した診断を出力する。
- **タスク**:
  1.  診断構造体 (コード、メッセージ、スパン、補足) の定義。
  2.  未定義/重複/型不一致/効果違反の基本コードを整備。
  3.  エラー回復の範囲 (継続可能なケース) を定義する。

## 3.7 検証
- **ユニット機能**:
  - 識別子解決: 使用箇所が宣言を指していることを確認。
  - 型チェック: `1 + 1` は有効、`1 + "a"` は無効。
  - 効果解析: `@pure` 関数で `io` を検出したらエラー。
- **テスト**:
  - `tests/unit/test_sema.c`: モック AST を使用して型結果をアサート。
  - 統合: 有効/無効な Reml スニペットを実行し、終了コード/エラーメッセージを確認。
  - 診断: 代表的な診断のゴールデン (JSON/テキスト) を追加。

## 3.8 フェーズ3の最小サブセット（仕様差分の明文化）
**対象仕様**: `docs/spec/1-2-types-Inference.md` / `docs/spec/1-3-effects-safety.md`

### 型システム（最小サブセット）
- **プリミティブ**: `i64`, `f64`, `Bool`, `Char`, `String`, `Unit` を扱う。
- **合成型**: `Tuple` と `Function` を実装対象に含める。
- **配列**: リテラルは **`[T]` のみ**を許可し、`[T; N]` は保留。
- **型推論**: ランク1の `let` 一般化、単一化、注釈の双方向検査を実装。
- **演算子/リテラル**: 数値リテラルは `i64`/`f64` のみ既定解決。演算子は組み込み扱いでトレイト解決は行わない。
- **保留**: ADT、レコード、参照/借用 (`&T`, `&mut T`)、型クラス/トレイト、`RunConfigTarget` 連携、パターン網羅性。

### 効果システム（最小サブセット）
- **効果タグ**: `mut`, `io`, `panic` のみ追跡。`ffi`/`unsafe`/`syscall` 等は解析対象外とし、使用時は未対応エラー。
- **属性**: `@pure` と `@no_panic` のみ検査する。その他属性は解析せずパース段階で通す。
- **値制限**: 効果を持つ `let` 束縛は単相とする（`io`/`mut`/`panic` が検出された場合）。
- **保留**: 効果行 (`! Σ`)、ハンドラ、`@cfg` 連携、Capability/Stage、監査タグ、`@dsl_export` の効果境界。
